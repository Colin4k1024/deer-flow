# 研究团队协作系统 - 需求简报

## 1. 背景与动机

### 现状分析

**已有实现**（证据）：
- `backend/research_team/` 目录已存在，包含完整的 Python 模块
  - `ResearchTeam` 编排器（`research_team.py`）：串行编排 5 个角色
  - 5 个角色 Agent：lead_researcher、information_collector、analyst、report_writer、reviewer
  - CLI 启动器（`run_research.py`）：支持 `--topic` + `--directions` 参数
  - `skills/public/deep-research/SKILL.md`：已配置 research methodology skill
- `backend/research_outputs/final_report_73b44ade.md`：已有测试输出，证明核心流程可运行

**缺失部分**（需要新增）：
- 无用户发布研究任务的 UI 界面
- 无研究项目的管理、进度追踪功能
- 当前是单机 CLI 模式，无团队协作机制
- 无 Web UI 与研究团队的对接入口

### 需求来源

用户在 DeerFlow 平台上增加**研究团队协作**能力，使用户可以：
1. 通过 Web UI 发布研究任务
2. 研究团队自动协作完成
3. 可视化追踪研究进度和中间产出

---

## 2. 目标与成功标准

### 目标

**核心目标**：在 DeerFlow 平台上实现用户发布研究任务 → 系统自动组建研究团队 → 多 Agent 协作完成 → 用户获取研究报告的完整流程。

### 成功指标（KPI）

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 研究任务发布成功率 | ≥95% | UI 操作 + 后端日志 |
| 团队协作完成率 | ≥90% | 项目状态统计 |
| 报告产出率 | ≥85% | 最终报告生成率 |
| 用户满意度 | ≥4.0/5 | 反馈收集 |
| 研究任务完成时间 | ≤30min（标准课题） | 端到端计时 |

### 用户价值

1. **降低研究门槛**：用户无需了解 Agent 技术，只需描述研究主题
2. **提升研究效率**：多 Agent 并行工作，比单一 Agent 快 3-5 倍
3. **保障研究质量**：首席研究员把关 + 审核员质检

## 3. 用户故事

### 故事 1：发布研究任务

**角色**：普通用户

**场景**：用户需要了解某个专业领域的研究报告

**流程**：
1. 用户点击"新建研究任务"
2. 输入研究课题（如"2026年协作机器人发展趋势"）
3. 选择或自定义研究方向（如"技术趋势"、"市场分析"、"应用案例"）
4. 点击"开始研究"
5. 系统自动创建研究项目，显示研究进度

**验收标准**：
- [ ] 用户可在 3 步内提交研究任务
- [ ] 研究方向支持多选和自定义
- [ ] 提交后立即显示项目 ID 和预计完成时间

### 故事 2：追踪研究进度

**角色**：普通用户

**场景**：用户发布任务后想了解研究进展

**流程**：
1. 用户进入"我的研究"列表
2. 点击具体项目查看详情
3. 可看到：
   - 当前阶段（制定计划/信息搜集/分析中/撰写报告/审核中/已完成）
   - 各 Agent 的中间产出预览
   - 预计剩余时间

**验收标准**：
- [ ] 进度实时更新（每分钟刷新）
- [ ] 中间产出可点击查看详情
- [ ] 支持中断/取消研究任务

### 故事 3：获取最终报告

**角色**：普通用户

**场景**：研究完成后用户查看报告

**流程**：
1. 研究完成通知推送给用户
2. 用户点击查看完整报告
3. 报告支持：
   - 在线阅读（带目录导航）
   - 下载（Markdown/PDF）
   - 分享链接

**验收标准**：
- [ ] 报告结构完整，包含执行摘要到参考资料
- [ ] 支持 Markdown 格式导出
- [ ] 报告数据可溯源到具体搜集来源

### 故事 4：团队角色管理（未来扩展）

**角色**：管理员

**场景**：配置研究团队的 Agent 角色

**流程**：
1. 管理员进入"团队设置"
2. 可查看/编辑各角色的 Prompt
3. 可添加自定义角色（如"行业专家"）
4. 角色变更自动应用到新任务

**验收标准**：
- [ ] 支持角色 Prompt 的 CRUD
- [ ] 支持自定义新角色
- [ ] 角色变更不影响历史项目

## 4. 范围定义

### In Scope（本次）

1. **后端 - 研究团队编排器增强**
   - 重构 `ResearchTeam` 支持异步任务队列
   - 增加项目状态持久化（JSON 文件或 SQLite）
   - 支持任务中断和恢复
   - 增加 Webhook/Callback 机制通知前端

2. **后端 - API 路由**
   - `POST /api/research/projects` - 创建研究项目
   - `GET /api/research/projects` - 列表用户项目
   - `GET /api/research/projects/{id}` - 项目详情
   - `DELETE /api/research/projects/{id}` - 取消项目
   - `GET /api/research/projects/{id}/outputs` - 中间产出
   - `GET /api/research/projects/{id}/report` - 最终报告

3. **前端 - 研究任务 UI**
   - 研究任务创建页面（表单）
   - 我的研究列表页面
   - 研究项目详情页（进度 + 中间产出）
   - 报告查看器

4. **集成测试**
   - 端到端研究任务流程测试
   - 多任务并行测试
   - 异常流程测试（超时、失败、取消）

### Out of Scope（本次不做）

1. 团队角色管理界面（留待未来迭代）
2. 报告版本管理和对比
3. 研究任务模板市场
4. 多团队/组织隔离
5. 研究报告评论和协作编辑
6. 实时流式输出（WebSocket）

## 5. 风险与依赖

### 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 多 Agent 并发调用 DeerFlowClient 稳定性 | 高 | 当前 `research_team.py` 使用 ThreadPoolExecutor，需验证连接池和线程安全 |
| 长时任务（30min+）超时 | 中 | 增加任务心跳机制，支持断点续传 |
| 研究报告版权/引用准确性 | 中 | 要求所有数据标注来源，审核员角色校验 |
| 模型调用成本控制 | 中 | 设置每个角色的最大 token 上限 |

### 依赖项

| 依赖 | 状态 | 说明 |
|------|------|------|
| DeerFlowClient | 已可用 | `packages/harness/deerflow/client.py` |
| deep-research skill | 已存在 | `skills/public/deep-research/SKILL.md` |
| research_team 模块 | 已存在 | `backend/research_team/` |
| 前端 Next.js 项目 | 已存在 | 可复用现有组件库 |
| Gateway API 框架 | 已存在 | FastAPI，可直接扩展 |

### 待确认项（已确认）

| # | 问题 | 决策 | 依据 |
|---|------|------|------|
| 1 | 数据持久化方案 | **SQLite** | 轻量、无需额外服务、开发友好 |
| 2 | 多用户隔离 | **需要** | 确保用户只能访问自己的研究项目 |
| 3 | 任务队列 | 不需要 | 当前 ThreadPoolExecutor 可满足需求 |
| 4 | 通知机制 | **轮询** | 简化实现，前端定期拉取状态 |
| 5 | 报告存储 | **本地存储** | `backend/research_outputs/` 目录 |

## 7. UI/UX 范围与质量门禁

### 目标端

- **主要**：Web 浏览器（桌面端优先）
- **次要**：移动端响应式（查看报告）

### 关键页面

1. **研究任务创建页** (`/research/new`)
   - 输入：课题名称（必填，≤100字）、研究方向（多选+自定义，≤5个）
   - 操作：开始研究按钮
   - 状态：loading → 跳转详情页

2. **研究列表页** (`/research`)
   - 列表：项目卡片（课题、状态、创建时间、进度%）
   - 操作：查看详情、取消
   - 状态：空态引导文案

3. **研究详情页** (`/research/{id}`)
   - 信息：项目 ID、状态、时间线
   - 阶段展示：计划 → 搜集 → 分析 → 撰写 → 审核 → 完成
   - 中间产出：可折叠的 Agent 输出面板
   - 最终报告：Markdown 渲染 + 下载按钮

4. **报告查看页** (`/research/{id}/report`)
   - 渲染：Markdown → HTML（带目录导航）
   - 操作：复制、下载、分享

### 质量门禁

| 门禁项 | 标准 |
|--------|------|
| 响应式 | 桌面 1440px +，平板 768px+，移动 375px+ |
| 可访问性 | 键盘导航、屏幕阅读器支持、颜色对比度 4.5:1 |
| 性能 | 页面首屏 <2s，API 响应 <500ms |
| 错误处理 | 所有异常有用户友好的错误提示 |

## 8. 候选动态分组（需求挑战会）

### 分组建议

基于技能和依赖关系，建议分为 3 个小组：

**分组 A：后端 - 研究编排器**
- 主责：`backend-engineer`
- 输入：现有 `ResearchTeam` 代码、`DeerFlowClient`
- 输出：增强的异步任务系统 + REST API
- 依赖：无外部依赖

**分组 B：前端 - 研究 UI**
- 主责：`frontend-engineer`
- 输入：设计稿（如需要）、后端 API 契约
- 输出：研究任务创建页、列表页、详情页
- 依赖：后端 API 完成后对接

**分组 C：集成测试**
- 主责：`qa-engineer`
- 输入：功能需求、API 契约
- 输出：E2E 测试用例
- 依赖：前后端集成完成

### 跨组协调点

1. API 契约需在设计阶段三方评审
2. 研究阶段定义（用于进度同步）需统一
3. 报告格式模板需前后端对齐

### 工作流程（现有）

```
1. 首席研究员制定计划
      ↓
2. 并行：信息搜集员 × N 个方向
      ↓
3. 并行：分析师 × N 个方向
      ↓
4. 报告撰写员整合 + 审核员审核
      ↓
5. 最终报告
```
